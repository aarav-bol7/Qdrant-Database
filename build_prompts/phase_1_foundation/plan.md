## 1. Plan summary

Build the project skeleton + infra layer + Docker Compose stack such that `docker compose up` brings six services online (postgres, redis, qdrant, web, grpc, worker) and `/healthz` returns a structured 200/503 JSON body reflecting Postgres + Qdrant reachability. Only `apps/core` is "real"; the other five apps are AppConfig-only stubs that future phases will fill in. The riskiest part is the Dockerfile + Compose handshake — chicken-and-egg with `uv sync --frozen` on first build, the web container's startup sequence (`migrate` → `collectstatic` → `gunicorn`) racing the Postgres healthcheck, and structlog cleanly intercepting Django + DRF + gunicorn stdlib loggers without dropping records or double-emitting. The build verifies itself end-to-end via the 10 acceptance criteria in the spec — combining `uv run pytest`, `docker compose ps` health states, `curl /healthz` (both green and Qdrant-stopped), `curl /admin/login/`, and `scripts/verify_setup.py`. Reproducibility is proven by `docker compose down -v && docker compose up -d --build` re-passing all 10.

## 6. Spec ambiguities & open questions

10 entries. All resolved here without user input.

### 6.1. structlog binding style

- **Ambiguous:** "Wire stdlib logging to delegate to structlog."
- **Decision:** Use `structlog.stdlib.ProcessorFormatter` via `dictConfig`. Loose `logging.getLogger(...)` calls flow through. Explicit handlers for `gunicorn.access`, `gunicorn.error`, `django`, `django.request`, `django.server`. Set `LOGGING_CONFIG = None` in `settings.py` so Django doesn't override.
- **Reversibility:** Easy — single file refactor.

### 6.2. `migrate --noinput` runs on web boot but Phase 1 has no models

- **Ambiguous:** Compose `web.command` runs `migrate` before gunicorn; Phase 1 has zero app-level models.
- **Decision:** Yes — Django's contrib apps (auth, admin, sessions, contenttypes, messages) own ~17 migrations that always run. They wire `auth_user` for `/admin/`, satisfying criterion 6. Don't stub `migrate`.
- **Reversibility:** Trivial. Migrations idempotent on subsequent boots; durable in `postgres_data`.

### 6.3. `tests/test_healthz.py` `@pytest.mark.django_db` — keep or drop?

- **Ambiguous:** Spec gives the test verbatim with the marker. Spec note "the test still passes because it accepts 503" implies infra-down tolerance.
- **Decision (revised in this version):** **Keep the marker verbatim per spec.** With it, pytest-django provisions a Postgres test DB; the healthz Postgres ping then succeeds inside the test. The 200/503 split exists for Qdrant being up vs down. Locally, devs `docker compose up -d postgres` before pytest; CI uses a service container.
- **Reversibility:** Trivial — one-line change.

### 6.4. `psycopg[binary]` runtime image: is `libpq5` actually needed?

- **Ambiguous:** psycopg's `[binary]` extra ships pre-built libpq inside the wheel; spec still installs `libpq5`.
- **Decision:** Keep `libpq5` per spec (~3 MB cost, future-proofing if the team switches to non-binary psycopg later).
- **Reversibility:** Trivial.

### 6.5. `STATIC_ROOT` directory existence before `collectstatic`

- **Decision:** `collectstatic --noinput` creates `STATIC_ROOT` if missing. No precondition needed. `.gitignore` excludes `staticfiles/`.
- **Reversibility:** Trivial — add `mkdir -p $STATIC_ROOT` if a flake appears.

### 6.6. `healthz` 2-second timeout — implementation choice

- **Decision:** `concurrent.futures.ThreadPoolExecutor(max_workers=2)` at module level; per-call `future.result(timeout=2)`. QdrantClient also takes `timeout=2` on construction; psycopg's `connect_timeout=2` set in `DATABASES.OPTIONS`. Belt + braces.
- **Reversibility:** Easy — single function.

### 6.7. Worker container dependencies — does it need Qdrant?

- **Decision:** No, per spec. Phase 1 worker is empty. Phase 5 will add `qdrant: {condition: service_healthy}` when async ingestion lands.
- **Reversibility:** Trivial — one YAML edit.

### 6.8. `redis` healthcheck on `web.depends_on` — needed?

- **Decision:** Honor spec — yes. `config/__init__.py` imports `celery_app`; module-level Celery doesn't connect at import. Adding redis to depends_on costs nothing and prevents Phase 5 surprises.
- **Reversibility:** Trivial.

### 6.9. `conftest.py` exact fixture surface

- **Decision:** Bare `client` fixture (override of pytest-django's auto-registered one is harmless). No DB fixtures.
- **Reversibility:** Trivial.

### 6.10. CI: `POSTGRES_HOST` for service containers

- **Decision:** In `ci.yml` job's `env:`, set `POSTGRES_HOST=localhost` and `QDRANT_HOST=localhost`. Compose-time `.env` is for local + production-style runs only.
- **Reversibility:** Trivial.

### Non-obvious flag choices

| Flag | Why |
|---|---|
| `uv sync` (no `--frozen`) on first run | lock doesn't exist; `--frozen` would fail. |
| `uv sync --frozen \|\| uv sync` in Dockerfile | Defensive — handles first-build (no lock) and locked rebuilds. |
| `--no-install-project --no-dev` (builder stage) | Skip project + dev deps to keep cache hot when source changes. |
| `--mount=type=cache,target=/root/.cache/uv` | Persistent uv cache; needs BuildKit (default in modern Docker). |
| `start_period: 30s` on web healthcheck | Buys time for `migrate` + `collectstatic` before failures count. |
| `prefer_grpc=True` on QdrantClient | Spec mandates; gRPC is faster. |
| `POSTGRES_HOST=localhost` in CI | Service containers expose ports on `localhost` from the runner; Compose hostnames don't apply. |
| `LOGGING_CONFIG = None` in settings | Stop Django from overriding our dictConfig. |
